## Architecture Overview

### Infrastructure Components

1. **Amazon EKS Cluster**
   - Kubernetes version: 1.32
   - Node groups: t3.medium (standard) or t3.small spot (cost-optimized)
   - Multi-AZ deployment capability
   - VPC with configurable NAT gateway

2. **Application Stack**
   - **n8n**: Latest version, workflow automation platform
   - **PostgreSQL 15**: Alpine-based, persistent data storage
   - **Storage**: AWS EBS gp3 volumes with encryption
   - **Networking**: Network Load Balancer for external access

3. **Operational Features**
   - Horizontal Pod Autoscaling (HPA)
   - Network policies for pod isolation
   - Automated daily backups via CronJob
   - CloudWatch integration for monitoring

### Deployment Patterns

**Standard Deployment**:
- 2x t3.medium nodes
- Persistent storage (EBS gp3)
- Production-ready configuration
- Estimated cost: ~$154/month (us-east-1)

**Cost-Optimized Deployment**:
- 1x t3.small spot instance
- Minimal resource allocation
- emptyDir storage (non-persistent)
- Estimated cost: ~$100/month
- **Use case**: Development/testing only

### Kubernetes Manifests (`manifests/`)

**Deployment Order** (numbered 00-11):

1. **00-namespace.yaml**: Creates `n8n` namespace with resource quotas
   - CPU: 2 requests, 4 limits
   - Memory: 4Gi requests, 8Gi limits
   - PVCs: 4 maximum

2. **01-postgres-secret.yaml**: Database credentials
   - ⚠️ **Security Note**: Hardcoded credentials (should use AWS Secrets Manager in production)
   - Default password: `n8n-secure-password-2024`

3. **02-persistent-volumes.yaml**: Storage claims
   - PostgreSQL: 20Gi gp3
   - n8n: 10Gi gp3
   - Access mode: ReadWriteOnce

4. **03-postgres-deployment.yaml**: PostgreSQL deployment
   - Image: postgres:15-alpine
   - Resources: 512Mi-1Gi memory, 250m-500m CPU
   - Health probes: liveness and readiness
   - Volume mount: /var/lib/postgresql/data

5. **04-postgres-service.yaml**: ClusterIP service
   - Internal access only (port 5432)
   - Service name: postgres-service-simple

6. **05-network-policy.yaml**: Network security
   - n8n ingress: port 5678 from any namespace
   - n8n egress: postgres (5432), HTTPS (443), HTTP (80), DNS (53)
   - postgres ingress: only from n8n pods
   - postgres egress: none (isolated)

7. **06-n8n-deployment.yaml**: n8n application
   - Image: n8nio/n8n:latest
   - Resources: 512Mi-1Gi memory, 250m-1000m CPU
   - Init container: volume permissions (busybox)
   - Environment: PostgreSQL connection, HTTP protocol
   - Health probes: /healthz endpoint

8. **07-n8n-service.yaml**: LoadBalancer service
   - Type: Network Load Balancer (NLB)
   - Internet-facing
   - Port mapping: 80 → 5678

9. **08-hpa.yaml**: Horizontal Pod Autoscaler
   - Min replicas: 1, Max: 5
   - CPU target: 70% utilization
   - Memory target: 80% utilization
   - Scale-down stabilization: 300s
   - Scale-up: immediate

10. **09-ingress.yaml**: ALB Ingress (optional)
    - Requires AWS Load Balancer Controller
    - HTTPS redirect capability
    - Custom domain support
    - Health check: /healthz

11. **10-backup-cronjob.yaml**: Automated backups
    - Schedule: Daily at 2 AM UTC
    - Retention: 7 days
    - Storage: 50Gi PVC
    - Optional S3 upload (commented out)

12. **11-restore-job.yaml**: Manual restore job
    - Finds latest backup automatically
    - Unzips and restores to PostgreSQL
    - Backoff limit: 3 attempts

### Cost Breakdown (us-east-1)

**Standard Deployment**:
- EKS Control Plane: $73/month
- EC2 (2x t3.medium): $58.40/month
- Network Load Balancer: $16.43/month
- EBS Storage (60GB): $6/month
- **Total**: ~$154/month

**Cost-Optimized Deployment**:
- EKS Control Plane: $73/month
- EC2 (1x t3.small spot): $8.76/month
- Network Load Balancer: $16.43/month
- EBS Storage (20GB): $1.60/month

### Cost Optimization Opportunities

1. **Scheduled Scaling**: Scale down during off-hours (40% savings)
2. **Reserved Instances**: 1-year commitment (30-40% savings)
3. **Fargate Alternative**: Serverless option ($80-120/month)
4. **ECS on EC2**: No EKS control plane fee ($50-80/month)

## Performance Considerations

### Current Configuration

**PostgreSQL**:
- CPU: 250m request, 500m limit
- Memory: 512Mi request, 1Gi limit
- Storage: 20Gi gp3

**n8n**:
- CPU: 250m request, 1000m limit
- Memory: 512Mi request, 1Gi limit
- Storage: 10Gi gp3

### Scaling Recommendations

**Light Workload** (< 100 workflows):
- Current configuration sufficient
- Consider cost-optimized deployment

**Medium Workload** (100-500 workflows):
- Increase PostgreSQL to 1Gi-2Gi memory
- Increase n8n to 1Gi-2Gi memory
- Enable HPA with min 2 replicas

**Heavy Workload** (> 500 workflows):
- Use managed RDS PostgreSQL
- Increase n8n to 2Gi-4Gi memory
- HPA with min 3, max 10 replicas
- Consider read replicas for PostgreSQL
